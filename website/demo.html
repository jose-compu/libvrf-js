EC VRF in your browser

Runs EC_VRF_P256_SHA256_TAI with WebCrypto. RSA suites need Node.js — see the home page.

What happens in one run?

A VRF takes a public input α and a secret key SK. Only the holder of SK can produce a proof π. Anyone with the public key PK can check π and learn the VRF output β — a deterministic pseudorandom value for that α.

Input α your message (UTF-8)
→

→
Proof π bytes you can send
→
verify() uses PK — anyone
→
Output β VRF value (hash)

Glossary — what is what?

Secret key (SK)

Private material generated in your browser. It never leaves this page unless you export it. Who can use it: only the prover — to call getVRFProofAsync.

Public key (PK)

Derived from SK. Safe to publish. Identifies which prover family you trust. Who can use it: verifiers — to call verifyVRFProofAsync.

Input α

Any byte string (here, text you type). Same α + same SK always yields the same β and π. Different α → different unpredictable β.

Proof π

Evidence that β was computed correctly from α under PK. Send π to a verifier along with α; they do not need SK.

VRF output β

The pseudorandom value returned after successful verification (from getVRFValue() on the proof). Behaves like a random oracle to outsiders who have not seen π.

This EC suite vs RFC 9381

This demo uses a simplified EC construction for browsers. Wire format and β differ from Microsoft libvrf / RFC 9381 ECVRF. RSA-FDH/PSS in the npm package are RFC 9381 aligned (Node only).
